## Web基础技术
### 1.url：统一资源定位符（网址）
http开头，所有页面都有唯一一个url
### 2.TCP/UDP：网络通信协议簇中的传输层
### 3.HTTP/HTTPS：网络通信协议簇中的应用层
我天天打交道的接口、网页请求，都基于这一层的协议。接口超时、请求失败，都和这层 + 传输层的规则有关。

TCP/UDP 是运货的货车（负责稳稳把数据送到目的地），HTTP/HTTPS 是货车里装货的装箱规范（规定货物怎么打包、收发格式），HTTP 跑在 TCP 的基础之上。
### 4.CDN/源服务器：从哪个节点拿资源
TCP：可靠传输（文件、网页、接口、CDN 静态资源）

CDN 的图片、录制好固定不变的缓存视频、JS、CSS、安装包，必须完整无损，全部依托TCP收发，丢包就重传，保证文件完整。

UDP：高速低延迟（CDN 实时音视频资源）

CDN 分发直播、短视频实时流，优先低延迟，容忍少量丢帧，走UDP传输。

CDN 不替代 TCP/UDP：CDN 只决定从哪个节点拿资源，会更快一些，而不改底层传输协议：就近节点和用户之间，依旧要么 TCP、要么 UDP。

### 5.前端技术
SDK：某个平台或服务为了方便开发者接入，提前打包好的一整套开发材料和工具。比如支付宝、微信支付、高德等等。

### 6.后端技术
同步：按顺序执行，调用某个东西时，调用方要等待这个调用返回结果才能继续往后执行

异步：调用方不会立即得到结果，调用发出后，调用方可以先执行后续操作，结果出来后再通知调用方

队列：先进先出，队尾进队头出，常用于秒杀、团购活动中

堆栈：后进先出

### 7.API接口
定义：系统之间数据交互的桥梁。通过API接口，可以获取一些数据或信息。

为什么需要API？

```text
解耦：客户端无需知道服务端的内部逻辑，只需按规则调用服务端的API，即可获取服务端的数据信息。
复用：一个服务端只需提供一套接口，即可被多个客户系统访问。比如支付宝API、高德API等。
安全：通过权限控制，限制访问范围，保护核心数据。
```
API的基本流程

```text
✅️请求（Request）-- 调用方发送指令，如“获取用户信息”
✅️处理（Process）-- 被调用方（服务端）验证权限、执行，比如查询数据库
✅️响应（Response）-- 返回结果，成功返回信息，或错误提示
```
**完整的 API 请求通常包括：请求方式 + URL + 请求参数 + 请求头/请求体（非必须，视请求方式而定，放在URL或者请求头体里都可以；后端相关，请求头负责描述这次请求，尤其是身份和格式；请求体负责承载真正提交给后端的数据）。**

调用 API 接口，就是调用方按照接口文档规定的请求方式、地址、参数、请求头/请求体，把请求发给被调用方服务器，服务器处理后再按约定格式返回结果或错误信息。

### 其他技术术语
1.写死：写死就是把值直接写固定了，不能灵活改。以后想改，就得改代码、重新上线。比如省市列表如果直接写在前端代码里，就是写死。页面只能显示代码里已有的省市。以后要新增城市，就必须改前端代码并重新发布。更好的做法是通过接口从后端获取省市列表，这样数据变化时不需要改前端代码。

2.开发说“做不了”，先搞清楚原因：
```text
现有的技术根本无法实现，也就是经常说的超出了技术边界，比如想要下载速度达到 10G/s
受限于公司研发能力和技术栈限制，超出了能力边界，比如原本是做小程序的要求去做人工智能开发
平台或生态的限制，不支持这种做法，或者没有开放对应的接口，比如微信没有对外开放获取好友列表的接口
现有技术架构不支持，如果要做成本会很高，需要重写底层代码，时间紧、任务重、不划算
实现方法比较复杂，没有快速实现的方法，无法满足短时间内实现的要求
觉得所提的需求价值不大，没有做的意义，也没有动力去做
单纯的不想做
烦你
```

3.任务难度判断：
**2.1看逻辑**
```text
• 逻辑本身很简单，没有很多的判断（比如只是增删改查），功能也比较独立，基本上评估 1~2 天是比较合理的，手脚快的可能半天就搞定了
• 如果知识纯展示，从几张表中拉去数据，加一些简单的判断，前端的复杂度和交互相关，后端复杂度较低，基本上也就半天一天的事
• 如果只是替换文字内容、图片、简单样式（颜色、大小等），哪怕页面比较多，也可以批量操作完成，可能繁琐但不复杂，可以在短时间内完成，属于举手之劳
```
**2.2 看资料：**
```text
• 一般有现成的接口文档和 SDK 的，实现起来也较为简单，花点时间研究下即可使用
• 有通用组件的，且之前有功能实现过，如果变化不大可以直接使用
```
**2.3 通用复杂度：**
```text
• 兼容性问题会比较复杂，特别是一些不常用机型的问题，有时还不好定位问题
• 性能问题通常比较难，需要一点点的去分析找到性能瓶颈，并测试优化
• 核心功能会比较复杂，尤其是要提供给很多模块使用的功能
• 研究性的工作通常比较耗时，产品经理在做需求时，如果发现可能需要用到新的技术，可以提前和开发打招呼，让他们先研究起来，把耗时问题先前置解决
```
